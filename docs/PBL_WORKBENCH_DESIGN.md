# Evolith PBL 工作台重设计方案

**版本**: v2.1 (实施对齐修订)
**日期**: 2026-04-27

---

## 一、设计哲学

> **核心转变**: 从"先学知识再做项目"变为"先做项目，卡壳时学知识"

当前 Evolith 是一个**知识图谱可视化工具**——学生看图谱、选知识点、生成课堂。
新设计是一个**PBL 项目工作台**——学生在项目沙盒中动手做，卡壳时 AI 渐进推送知识。

**关键变化**: 知识图谱保持完整展示（含项目节点、知识点节点、里程碑节点和所有边类型），学生可以浏览全局课程结构。同时新增 PBL 工作台作为核心学习界面——学生在工作台中按里程碑推进项目，卡壳时由 AI 主动推送知识。图谱和工作台互为补充：图谱提供全局视野，工作台提供项目实操。

## 二、新的用户旅程

### 当前旅程（问题）

```
看图谱 → 选知识点 → 生成虚拟课堂 → 学完 → 不知道下一步去哪里
→ 缺少项目驱动的实操体验，知识点是孤立的学习对象
```

### 新旅程

```
首页 → 构建课程，查看完整知识图谱（项目 + 知识点 + 里程碑）
  → 点击项目节点 → 详情面板出现"进入 PBL 学习"按钮
  → 进入项目工作台 → 按里程碑推进
  → 卡壳时主动点 💡求助 → 渐进式知识推送（提示 → 深度讲解 → 完整课堂）
  → 完成项目 → 回到图谱，进入下一个项目
```

## 三、核心组件设计

### 3.1 知识图谱页面 (Process.vue，保持现有完整展示)

**图谱展示保持不变**，继续展示完整的知识图谱结构：

```
显示所有节点和边:
- Project 节点 (圆角矩形)
- KnowledgePoint 节点 (圆形)
- Milestone 节点
- NEXT_STEP / REQUIRES / PREREQUISITE_OF / MILESTONE_STEP / MILESTONE_REQUIRES 边
- 两种视图模式（全图视图 / 学习路径视图）
```

图谱和工作台互为补充：
- **图谱**提供全局视野——课程整体结构、项目间关系、知识点依赖
- **工作台**提供项目实操——里程碑推进、知识推送、进度追踪

从图谱进入工作台的方式：点击项目节点 → 详情面板出现"进入 PBL 学习"按钮 → 跳转到 `/workbench/:projectId`。

### 3.2 项目工作台 (ProjectWorkbench)

**新增核心页面，与 Process.vue 图谱视图互补**

实际实现采用**双栏布局**（里程碑栏 + 工作区），知识面板为右侧抽屉覆盖而非常驻第三栏：

```
┌─────────────────────────────────────────────────────────┐
│ 项目工作台 - "实现评论系统"                              │
│                                                         │
│ ┌─────────┐ ┌──────────────────────────────────────────┐│
│ │里程碑栏 │ │   工作区                                 ││
│ │(左侧)   │ │   (主区域)                               ││
│ │         │ │                                          ││
│ │ ✓ 步骤1 │ │  当前里程碑的:                           ││
│ │ ✓ 步骤2 │ │  · 任务描述                              ││
│ │ ◐ 步骤3 │ │  · 验收标准                              ││
│ │ ○ 步骤4 │ │  · 关联知识点卡片 (含课堂按钮)           ││
│ │ ○ 步骤5 │ │  · 学习笔记输入区 (自动保存)             ││
│ │         │ │  · [标记完成] 按钮                        ││
│ │ 💡求助   │ │                                          ││
│ └─────────┘ └──────────────────────────────────────────┘│
│                                                         │
│ 点击 💡求助 或知识点卡片 → 右侧抽屉弹出 HintPanel       │
│ 点击"生成虚拟课堂"/"进入虚拟课堂" → 课堂生成/跳转       │
│                                                         │
│ 底部: 进度 2/5 │ 已用提示 0次 │ [回到项目路径]          │
└─────────────────────────────────────────────────────────┘
```

**关键交互**:

- **里程碑栏**: 显示当前进度，已完成步骤可回顾描述，当前步骤高亮
- **工作区**: 包含关联知识点卡片列表，每张卡片显示名称、难度徽章、摘要，以及课堂按钮：
  - 无 classroom_id → 橙色"生成虚拟课堂"按钮
  - 有 classroom_id → 绿色"进入虚拟课堂"按钮
- **知识面板**: HintPanel 右侧抽屉覆盖，学生点击 💡求助 或点击 KP 卡片时弹出
- **求助按钮**: 唯一的卡壳触发方式——学生主动点击，AI 不自动检测
- **学习笔记**: 文本输入区，自动保存到 localStorage，切换里程碑时恢复

### 3.3 里程碑系统 (Milestone)

每个项目分解为 3-5 个里程碑，**由 LLM 提取时一并生成**：

```
Project: "实现评论系统"
├── Milestone 1: "搭建数据模型"
│   ├── 任务: 设计评论的数据结构，包含用户名、内容、时间戳
│   ├── 验收: 提交数据模型描述，包含必填字段和类型
│   └── 关联知识点: [数据建模基础, 关系型数据库概念]
│
├── Milestone 2: "实现 CRUD API"
│   ├── 任务: 为评论实现增删改查四个接口
│   ├── 验收: 四个接口都能正确响应
│   └── 关联知识点: [RESTful API, CRUD操作, HTTP请求方法]
│
├── Milestone 3: "添加用户认证"
│   ├── 任务: 实现简单的用户登录，只有登录用户才能发评论
│   ├── 验收: 未登录用户无法POST，登录用户可以
│   └── 关联知识点: [身份认证基础, Session vs Token]
│
├── Milestone 4: "实现评论嵌套"
│   ├── 任务: 支持对评论的回复（二级嵌套）
│   ├── 验收: 可以对评论回复，回复显示在评论下方
│   └── 关联知识点: [树形数据结构, 递归查询]
│
└── Milestone 5: "优化与部署"
    ├── 任务: 添加分页功能，部署到服务器
    ├── 验收: 评论列表支持分页，服务可访问
    └── 关联知识点: [分页算法, 部署基础]
```

**关键点**: 每个里程碑绑定一组知识点（来自图谱中该项目的 REQUIRES 知识点）。
当学生求助时，AI 查当前里程碑关联的知识点，生成推送内容。

### 3.4 卡壳检测：仅主动求助

**设计原则**: 最简单、最尊重学生自主性。

- **唯一触发方式**: 学生点击 💡求助 按钮
- **不做**: 自动检测停留时间、AI对话分析、验收失败强制推送
- **理由**:
  - PBL 强调学生自主探索，自动推送会打断思考过程
  - 主动求助意味着学生已经尝试过，此时推送最有效
  - 实现简单，不需要复杂的检测逻辑

触发流程：
```
学生点击 💡求助
→ 确定: 当前里程碑 + 关联知识点
→ 生成 Level 1 提示 (LLM 实时调用，几秒延迟)
→ 显示在右侧抽屉面板
→ 学生选择: [明白了] / [还是不太懂]
→ 如果 [还是不太懂] → 生成 Level 2 讲解
→ 如果 [我要完整课堂] → Level 3 OpenMAIC
```

## 四、渐进式知识推送系统

### 三级推送设计

```
💡求助 触发
│
├── Level 1: 快速提示 (1-2分钟)
│   ├── 形式: 右侧抽屉面板，纯文字
│   ├── 内容: 2-3句话解释核心概念 + 一个小提示
│   ├── 按钮: [明白了，继续]  [还是不太懂 ↓]
│   ├── 生成: LLM 实时调用
│   └── 成本: 低（一次短文本调用）
│
├── Level 2: 深度讲解 (3-5分钟)
│   ├── 形式: 右侧抽屉扩展，文字 + 示例 + 常见错误
│   ├── 内容: 详细解释 + 与项目相关的具体示例 + 易犯错误
│   ├── 按钮: [回去试试]  [我要完整课堂 ↓]
│   ├── 生成: LLM 实时调用
│   └── 成本: 中
│
└── Level 3: 沉浸式课堂 (5-10分钟)
    ├── 形式: 全屏 OpenMAIC 虚拟课堂（iframe）
    ├── 内容: 完整的结构化课堂，幻灯片+语音+测验
    ├── 退出: 课堂结束后 [回到项目工作台]
    ├── 生成: OpenMAIC 生成（现有流程）
    ├── 缓存: classroom_id 写入 Neo4j，下次直接进入
    ├── 状态: KP 卡片按钮自动从"生成课堂"变为"进入课堂"
    └── 成本: 高
```

### 提示生成 Prompt

**Level 1 提示**:

```python
HINT_L1_PROMPT = """
你是一个 PBL 学习助手。学生在做项目时卡壳了，主动求助。

项目: {project_name}
项目目标: {project_outcomes}
当前里程碑: {milestone_name}
里程碑任务: {milestone_description}
关联知识点: {kp_name}
知识点摘要: 学习前需要掌握 {kp_prerequisites}，学完后能 {kp_outcomes}

给一个简短的提示帮助学生继续:
- 2-3句话，直接点明核心概念
- 不展开成教程，只给点拨
- 结尾给一个具体的小提示让学生可以继续动手
- 语言简洁，1-2分钟可看完
"""
```

**Level 2 讲解**:

```python
HINT_L2_PROMPT = """
你是一个 PBL 学习助手。学生需要更详细的讲解才能继续项目。

项目: {project_name}
当前里程碑: {milestone_name}
关联知识点: {kp_name}
知识点摘要: {kp_prerequisites} → {kp_outcomes}

给一个深度讲解:
- 详细解释核心概念（3-5段）
- 给出与当前项目 {project_name} 直接相关的具体示例
- 列出2-3个学生容易犯的错误
- 每段结尾给出实践建议
- 总长度3-5分钟阅读量
"""
```

## 七、LLM 提取 Prompt 变更

### 变化要点

当前 prompt 要求 LLM 只提取项目和知识点，不提取里程碑。
新 prompt 要求 LLM 一并提取里程碑分解。

不再要求 LLM 构建环形闭合——环形由里程碑顺序自然完成：

```
提取内容 (4 层):

1. 项目序列 (NEXT_STEP 主路径)         — 不变
2. 每个项目的知识点 (REQUIRES)          — 不变
3. 知识点前置依赖 (PREREQUISITE_OF)     — 不变
4. 每个项目的里程碑 (MILESTONE_STEP)    — 新增
   - 每个里程碑的名称、描述、验收标准
   - 每个里程碑关联哪些知识点 (MILESTONE_REQUIRES)

环形闭合简化:
- 不再要求 "最后一个 KP → 同一个 Project" 的 REQUIRES 边
- 环形 = 里程碑全部完成 → 项目完成 → 进入 NEXT_STEP 项目
- 知识点仍然有 REQUIRES 和 PREREQUISITE_OF 关系，但不再需要闭环约束
- 实现保留了向后兼容：旧项目的环形 REQUIRES 边仍会被读取和展示
```

这样 LLM 只需要做它擅长的事（分解项目步骤、标注知识依赖），不做它不擅长的事（维护环形结构）。

```json
            "order": 2,
            "knowledge_point_uuids": ["kp_3"]
          }
        ]
      }
    },
    {
      "label": "KnowledgePoint",
      "uuid": "kp_1",
      "name": "HTML基础",
      "attributes": { ... }
    }
  ],
  "edges": [
    { "source": "proj_1", "target": "proj_2", "type": "NEXT_STEP" },
    { "source": "proj_1", "target": "kp_1", "type": "REQUIRES" },
    { "source": "kp_1", "target": "kp_2", "type": "PREREQUISITE_OF" }
  ]
}
```

里程碑作为 Project 节点 attributes 的嵌套数据输出，而不是独立节点。
存入 Neo4j 时，后端将里程碑拆分为独立 Milestone 节点 + MILESTONE_STEP/MILESTONE_REQUIRES 边。

## 八、实施阶段

### Phase 1: MVP（核心骨架）

必须实现:

1. **LLM 提取 prompt 改造** — 输出里程碑数据
2. **Neo4j 数据模型扩展** — Milestone 节点 + 新边类型
3. **项目工作台页面** — ProjectWorkbench.vue (里程碑栏 + 工作区 + 求助按钮)
5. **渐进式知识推送** — HintPanel.vue + HintService (L1/L2 LLM实时生成)
6. **L3 课堂集成** — 复用现有 OpenMAIC 流程，从工作台弹出
7. **学生进度追踪** — localStorage 存储
8. **里程碑验收** — 学生自评标记完成（最简单方式）

可以延后:

- 代码编辑器/实际工作区（先用文本提交）
- LLM 自动评估验收（先用自评）
- 登录系统（先用 localStorage）
- 预缓存 L1 提示（先用实时生成）

### Phase 2: 工作区 + 验收

- 根据项目类型提供工作区（代码编辑器 / 文档编辑器 / 画布）
- LLM 验收里程碑提交（评估是否达标 + 反馈）
- Level 1 提示预缓存（提取时生成，推送时无延迟）
- 简单登录系统（进度持久化到后端）

### Phase 3: 学习分析 + 完善

- 学生学习数据可视化（哪些知识点是高频求助点）
- 教师/管理员仪表板
- 时间停滞检测（可选的自动提示）
- 项目完成 AI 评估报告

## 十、关键决策记录

| 决策 | 结论 | 理由 |
|------|------|------|
| D1 工作区 | Phase 1 文本提交 | 最简单，任何项目类型通用 |
| D2 验收 | Phase 1 学生自评 | 不需要 LLM 评估逻辑 |
| D3 身份 | Phase 1 无登录, localStorage | 零门槛 |
| D4 里程碑来源 | LLM 提取时一并生成 | 自动化，不需要人工配置 |
| D5 提示生成 | 实时生成 | 延迟几秒可接受，可结合上下文更精准 |
| D6 卡壳检测 | 仅主动求助 | 最简单、最尊重学生自主性 |
| D7 图谱展示 | 保持完整展示，不简化 | 图谱和工作台互为补充：图谱提供全局视野，工作台提供项目实操。两种视图满足不同需求 |
| D8 classroom_id 缓存 | 课堂生成后异步写入 Neo4j | Fire-and-forget 方式，不阻塞课堂导航。写入有 2 次重试应对 AuraDB 连接断开。缓存成功后图谱和工作台自动显示"进入课堂"按钮 |
| D9 课堂轮询 | 无超时上限 | PBL 场景课堂生成可能较长，用 setTimeout 递归轮询替代 setInterval+maxAttempts |
